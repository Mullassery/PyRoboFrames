## Phase 1: Video Codec Selection (Week 3-4, Task #6)

### Goals

- Measure storage savings (expected 40-50% for HEVC)
- Document trade-offs (encode speed, decode speed, device compatibility)

### Implementation Details

#### 1.1 Schema Changes

**LeRobot Meta Format Extension (meta/info.json):**
```json
{
  "codebase_version": "v3.0",
  "fps": 30,
  "video_codec": "hevc",  // NEW: "h264" (default), "hevc", "av1"
  "video_profile": "main",  // h264: baseline, main, high; hevc: main, main10
  "features": {
    "observation.images.top": {
      "dtype": "video",
      "shape": [480, 640, 3],
      "codec": "hevc"  // Can override per-camera
    }
  }
}
```

#### 1.2 API Design

**Writing Datasets with Codec Selection:**
```python
prf.write_lerobot_dataset(
    "my_dataset/",
    features={...},
    episode_lengths=[...],
    fps=30.0,
    video_codec="hevc",        # NEW: "h264" (default), "hevc", "av1"
    video_profile="main",      # hevc: main (8-bit), main10 (10-bit)
    encoder_preset="medium",   # ffmpeg preset: ultrafast...veryslow
)
```

**Loading with Codec-Specific Options:**
```python
ds = prf.RoboFrameDataset.from_path("my_dataset/")
loader = ds.loader(
    cameras=["observation.images.top"],
    codec="auto",  # NEW: auto-detect from meta/info.json, or override
    hardware_decode=True,  # Use NVDEC/VideoToolbox if available
)
```

#### 1.3 Storage Format

**Codec metadata in dataset:**
- Store in `meta/info.json` (global default)
- Per-camera override in feature metadata
- Add to episodes metadata: `video_codec` per episode (for mixed-codec datasets)
- Add version to info: `video_codec_version: 1`

#### 1.5 Codec-Specific Considerations

**H.264 (Default)**
- Pros: Universal hardware support, fastest decode, smallest file sizes for modern hardware
- Cons: Patents (licensing concerns in some regions)
- FFmpeg encoders: libx264, h264_nvenc, h264_videotoolbox
- Profile: baseline (streaming), main (compatible), high (quality)

**HEVC/H.265**
- Pros: 40-50% better compression than H.264, good hardware support (RTX 20+, Apple T2+)
- Cons: Slower encoding, some older devices lack support
- FFmpeg encoders: libx265, hevc_nvenc, hevc_videotoolbox
- Profile: main (8-bit), main10 (10-bit, higher quality)

**AV1**
- Pros: Best compression (30-40% better than HEVC), royalty-free, future-proof
- Cons: Very slow encoding, limited hardware decode support (RTX 50xx+, recent Apple)
- FFmpeg encoders: libaom-av1, av1_nvenc (RTX 50xx+)
- Profile: main

## Phase 2: Depth Camera Support (Week 5-7, Task #7)

### Goals
- Add point cloud reading (Oak-D, RealSense, generic formats)
- Time-align depth with RGB frames
- Extend LeRobot format for depth data
- Memory-mapped storage for efficiency

### Key Design Decisions

**Point Cloud Formats:**
- `.xyz`: Simple text (x, y, z per line)
- `.ply`: ASCII/binary (Polygon File Format)
- `.pcd`: Binary (Point Cloud Data, ROS standard)
- Raw: NumPy `.npy` (H×W×3 depth maps)

**Time Alignment Strategy:**
- Use existing `delta_timestamps` mechanism
- Depth timestamp = RGB timestamp + offset
- Handle asynchronous camera timing (depth lags RGB by ~50ms typical)

**Storage:**
- Video codecs for RGB frames (already in HEVC/H.264)
- Raw depth: memory-mapped `.mmap` files or `.npz` for compression
- Optional depth compression (PNG 16-bit, TIFF, or lossless WebP)
